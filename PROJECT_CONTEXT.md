## Tech Stack & Dependencies
- **Language**: Java 21
- **Framework**: Spring Boot
- **Database**: PostgreSQL (configured for Neon)
- **Database Migrations**: Flyway
- **ORM**: Spring Data JPA
- **Caching & Rate Limiting**: Spring Data Redis (configured for Upstash)
- **Security**: Spring Security
- **Authentication**: JWT (using `jjwt` v0.12.3)
- **File Storage**: Backblaze B2 (using AWS S3 SDK `software.amazon.awssdk:s3:2.25.0`)
- **API Documentation**: Springdoc OpenAPI UI (Swagger)
- **Other**: Lombok, Validation, Spring Web, Spring WebSocket

## Environment Configuration
The application relies on the following environment variables (configured in `application.properties` and read from `.env`):
- **Database**: `DATABASE_URL`, `DATABASE_USERNAME`, `DATABASE_PASSWORD`
- **Redis**: `REDIS_URL`
- **Security**: `JWT_SECRET`
- **Backblaze B2**: `B2_ENDPOINT`, `B2_REGION`, `B2_KEY_ID`, `B2_APP_KEY`, `B2_BUCKET`
- **Frontend**: `FRONTEND_URL` (defaults to `http://localhost:5173`)

## Architecture Plan Highlights
- **Sample Library Extensibility**: The `rooms` table utilizes a `battle_config` JSONB column. This ensures the app can easily extend from Beat Battles to Art Battles without altering the database schema.
- **Direct File Uploads**: The frontend will upload files directly to Backblaze B2 using Presigned URLs generated by the backend, saving server bandwidth.
- **State Machine**: Rooms transition automatically through `WAITING -> ACTIVE -> VOTING -> CLOSED` using Spring `@Scheduled` tasks and Optimistic Locking (`@Version`).
